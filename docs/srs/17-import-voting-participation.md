# Импорт участия в голосовании ОСС

**Порядок: LOST-01, CORE-01 → CORE-05 → ADM-10**

Фича даёт суперадмину возможность загружать из Excel/CSV данные об участии собственников в голосовании ОСС: кадастровый номер помещения и доля в собственности. Данные хранятся отдельно от контактов и анкет (`contacts`, `oss_voting`). На первом этапе импорт **не влияет** на расчёт кворума (CORE-04).

---

## CORE-05 — Импорт участия в голосовании (CSV/XLS/XLSX)

### 1. Архитектурный анализ (CoT)

Импорт выполняется в Backend (FastAPI), данные сохраняются в PostgreSQL. Используются те же механизмы парсинга файлов, что и в CORE-01/ADM-06: чтение CSV/XLS/XLSX, маппинг колонок по алиасам ([import_register.py](../../backend/app/import_register.py)). **Отличие от ADM-06:** в файле нет ПДн — шифрование BE-02 не применяется; данные привязаны к помещению, а не к контакту; при каждой успешной загрузке таблица `oss_participation` **полностью перезаписывается**. Доступ — только **super_administrator** (как CORE-01). Зависимости: LOST-01 (схема БД, таблица `oss_participation`), CORE-01 (реестр помещений в `premises`), ADM-01 (авторизация), BE-03 (аудит).

### 2. Описание и цель

Инструмент загрузки **фактов участия в голосовании ОСС** из файлов CSV/XLS/XLSX. Одна строка файла = один собственник с указанной долей в собственности по помещению с известным кадастровым номером. Позиция по голосованию («ЗА»/«Против») в файле **отсутствует** — импортируется только факт участия. Цель — зафиксировать в системе данные из протокола/выгрузки голосования для последующего анализа суперадмином (без изменения публичного кворума на первом этапе).

### 3. Функциональные требования

* **SR-CORE05-001:** Система должна содержать миграцию Alembic, создающую таблицу `oss_participation` (участие_в_голосовании) с полями: `id` (serial PK), `premise_id` (FK → `premises.cadastral_number`), `share_nominal` (NUMERIC, номинал доли из файла), `ownership_share` (NUMERIC, суммарная доля по группе), `participated` (boolean, default true), `imported_at` (timestamptz), `import_batch_id` (uuid, опционально). Уникальное ограничение на пару `(premise_id, share_nominal)`.
* **SR-CORE05-002:** Система должна принимать файл загрузки участия через отдельный API (доступ только для роли super_administrator).
* **SR-CORE05-003:** Система должна поддерживать форматы CSV (UTF-8), XLS и XLSX в соответствии с CORE-01.
* **SR-CORE05-004:** Обязательными для файла являются колонки: кадастровый номер (`cadastral_number` или алиас) и доля в собственности (`ownership_share` или алиас). При отсутствии любой из обязательных колонок система должна возвращать ошибку структуры (400) с `expected_columns` и `detected_columns`; импорт не выполняется, таблица `oss_participation` не изменяется.
* **SR-CORE05-005:** Система должна обрабатывать только строки, в которых помещение с указанным кадастровым номером **уже существует** в таблице `premises`. Строки с неизвестным кадастровым номером не импортируются; в отчёт включается ошибка по номеру строки (например, «Premise not found»).
* **SR-CORE05-006:** Система должна нормализовать значение доли в собственности в диапазон (0, 1]: поддерживаемые форматы — десятичная дробь (`0.5`), обыкновенная дробь (`1/2`, `1/3`), процент (`50%`). При невалидном формате или значении ≤ 0 или > 1 строка отклоняется с сообщением в `errors`.
* **SR-CORE05-007:** При совпадении пары (кадастровый номер + доля в собственности) в нескольких строках одного файла система должна **агрегировать** их: значения доли суммируются, в БД записывается одна строка с итоговой суммой. Все исходные строки группы считаются принятыми (`accepted`).
* **SR-CORE05-008:** При каждой успешной загрузке (файл распарсен, обязательные колонки найдены) система в рамках одной транзакции должна сначала удалить **все** строки таблицы `oss_participation`, затем вставить только валидные строки нового файла. Актуальным остаётся только последний успешный импорт.
* **SR-CORE05-009:** Невалидные строки не записываются в БД; для каждой — запись в `errors` с номером строки и кратким сообщением. Валидные строки при этом сохраняются (частичный импорт, как CORE-01/ADM-06).
* **SR-CORE05-010:** Система должна возвращать отчёт в том же формате, что и CORE-01: `accepted`, `rejected`, `errors` (массив `{ row, message }`).
* **SR-CORE05-011:** При критической ошибке (формат файла, недоступность БД) система откатывает транзакцию и не вносит изменений в `oss_participation`.
* **SR-CORE05-012:** Система должна записывать в аудит-лог (BE-03) факт импорта: действие `import_voting_participation`, идентификатор администратора, количество `accepted` и `rejected`, `import_batch_id` (без ПДн — их в файле нет).
* **SR-CORE05-013:** 🤔 При сумме долей по одному кадастровому номеру > 1 в рамках одного файла (после агрегации по SR-CORE05-007) система должна добавлять в отчёт предупреждение (поле `warnings`, не блокирующее импорт валидных строк).

### 4. Сценарий использования

**Триггер:** Суперадмин загружает файл с данными об участии в голосовании (кадастровый номер + доля в собственности).

**Main Success Scenario:**
1. Суперадмин выбирает файл CSV/XLS/XLSX с обязательными колонками и отправляет запрос на импорт.
2. Система проверяет права (super_administrator), парсит файл, проверяет наличие обязательных колонок.
3. Система в транзакции удаляет все записи `oss_participation`, валидирует каждую строку, агрегирует строки с одинаковой парой кадастр+доля (суммирование долей), вставляет итоговые записи, фиксирует аудит и возвращает отчёт (`accepted`, `rejected`, `errors`).

**Alternative Flows:**
* **AF-1 (Нет обязательных колонок):** Система возвращает 400 с `expected_columns` и `detected_columns`; таблица не меняется.
* **AF-2 (Помещение не найдено):** Строка отклоняется, в `errors` — номер строки и сообщение.
* **AF-3 (Невалидная доля):** Строка отклоняется, в `errors` — причина (формат, диапазон).
* **AF-4 (Одинаковая пара кадастр+доля):** Строки агрегируются: доли суммируются, в БД — одна запись с итоговой суммой; все строки группы учитываются в `accepted`.
* **AF-5 (Все строки отклонены):** Транзакция завершается: таблица `oss_participation` очищена, новых записей нет; в отчёте `accepted: 0`, `rejected: N`.
* **AF-6 (403 для обычного админа):** Роль administrator получает 403; UI блока загрузки скрыт (ADM-10).

### 5. Модель данных

**Новая таблица `oss_participation` (участие_в_голосовании):**

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | serial PK | |
| `premise_id` | string FK → `premises.cadastral_number` | Кадастровый номер помещения |
| `share_nominal` | NUMERIC(10,6) | Номинал доли из файла (ключ группы при агрегации) |
| `ownership_share` | NUMERIC(10,6) | Суммарная доля по группе (SR-CORE05-007) |
| `participated` | boolean, default true | Факт участия в голосовании |
| `imported_at` | timestamptz | Время импорта строки |
| `import_batch_id` | uuid, nullable | Идентификатор пакета загрузки |

**Связи:** `premises` → `oss_participation` (одно помещение — несколько записей при разных долях; строки файла с одинаковой парой кадастр+доля агрегируются суммированием долей — SR-CORE05-007). Связи с `contacts` и `oss_voting` **нет** — это независимый снимок данных из внешней выгрузки.

**Обязательные колонки файла:** `cadastral_number`, `ownership_share`.

**Алиасы заголовков колонок:**

| Каноническое имя | Допустимые алиасы (в т.ч. русские) |
|------------------|-------------------------------------|
| cadastral_number | кадастровый_номер, кадастровый номер |
| ownership_share | доля_в_собственности, доля в собственности, доля |

**Парсинг значения доли:**

| Формат в файле | Пример | Нормализованное значение |
|----------------|--------|--------------------------|
| Десятичная дробь | `0.5`, `0,5` | `0.5` |
| Обыкновенная дробь | `1/2`, `1/3` | `0.5`, `0.333333` |
| Процент | `50%`, `50 %` | `0.5` |
| Целое (доля = 1) | `1` | `1.0` |

### 6. Интеграции и API

* **Method:** `POST /api/admin/import/voting-participation`
* **Request:** multipart/form-data с полем `file`. Заголовок `Authorization: Bearer <JWT>`. Доступ — только роль `super_administrator` (с учётом согласия с Политикой, ADM-09).
* **Response (успех):**
```json
{
  "accepted": 120,
  "rejected": 3,
  "errors": [
    { "row": 5, "message": "Premise not found: 77:02:0015008:9999" },
    { "row": 12, "message": "Invalid ownership share: abc" }
  ],
  "warnings": [
    { "cadastral_number": "77:02:0015008:1234", "message": "Sum of shares exceeds 1.0: 1.25" }
  ]
}
```
Поле `warnings` — по SR-CORE05-013; при отсутствии предупреждений может быть пустым массивом или опущено.

* **Response (ошибка структуры, 400):**
```json
{
  "detail": "Required columns missing: cadastral_number, ownership_share",
  "expected_columns": ["cadastral_number", "ownership_share"],
  "detected_columns": ["cadastral_number", "area"]
}
```

### 7. Нефункциональные требования

* **Security:** Доступ только для super_administrator (ADM-01); логирование факта импорта (BE-03). ПДн в файле отсутствуют — шифрование не требуется.
* **Performance:** Рекомендуемый лимит размера файла тот же, что для CORE-01 (например, до 5 MB).
* **Scope:** Визуализация участия реализована в FE-06 (шахматка, блок «Статистика ОСС») и SR-CORE04-008. Расчёт «ЗА» по `oss_voting` в API кворума сохранён для обратной совместимости, на главной не отображается.

## ADM-10 — UI суперадмина: загрузка участия в голосовании

### 1. Архитектурный анализ (CoT)

Реализуется на Frontend (React) в составе страницы загрузки (LOST-02). Взаимодействует с `POST /api/admin/import/voting-participation`. Блок виден **только** суперадмину (аналогично загрузке реестра в LOST-02); для роли administrator скрыт. Зависимости: CORE-05, LOST-02, ADM-01, ADM-09.

### 2. Описание и цель

Интерфейс выбора файла и загрузки данных об участии в голосовании ОСС: инструкция по формату и обязательным колонкам, предупреждение о полной замене данных при новой загрузке, отчёт по строкам. Визуализация загруженных данных — на главной (FE-06, SR-CORE04-008).

### 3. Функциональные требования

* **SR-ADM10-001:** Система должна отображать блок «Загрузка участия в голосовании» **только** для роли `super_administrator` на странице загрузки ([Upload.jsx](../../frontend/src/pages/Upload.jsx)). Для роли `administrator` блок не отображается.
* **SR-ADM10-002:** Блок должен содержать инструкцию: поддерживаемые форматы (.csv, .xlsx, .xls), обязательные колонки («Кадастровый номер», «Доля в собственности»), примеры значений доли (`1/2`, `0.333`, `50%`), пояснение что помещения должны уже быть в реестре (CORE-01), а строки с одинаковой парой кадастр+доля суммируются (SR-CORE05-007).
* **SR-ADM10-003:** Блок должен содержать явное предупреждение: **новый файл полностью заменяет** все ранее загруженные данные участия в голосовании.
* **SR-ADM10-004:** Элемент выбора файла и кнопка отправки должны быть доступны с клавиатуры и с мобильных устройств (аналогично LOST-02).
* **SR-ADM10-005:** При ошибке структуры (400 с `expected_columns`/`detected_columns`) система должна отображать сравнение «Обнаруженные колонки» и «Ожидаемые колонки» (аналогично LOST-02 SR-LOST02-005).
* **SR-ADM10-006:** После успешной загрузки система должна отображать отчёт: количество принятых и отклонённых строк, при наличии — раскрываемый список ошибок по номерам строк.
* **SR-ADM10-007:** При наличии предупреждений (`warnings` в ответе API) система должна отображать их отдельным блоком (не блокирующим успех импорта).

### 4. Сценарий использования

**Триггер:** Суперадмин открывает страницу загрузки и выбирает загрузку участия в голосовании.

**Main Success Scenario:**
1. Суперадмин видит блок «Загрузка участия в голосовании» с инструкцией и предупреждением о полной замене данных.
2. Суперадмин выбирает файл с колонками «Кадастровый номер» и «Доля в собственности» и нажимает отправку.
3. Система отправляет файл на API, получает отчёт и отображает результат (`accepted`, `rejected`, список ошибок, предупреждения).

**Alternative Flows:**
* **AF-1 (Обычный администратор):** Блок не отображается; прямой вызов API — 403.
* **AF-2 (Ошибка структуры):** Отображение сравнения колонок; таблица `oss_participation` не изменяется.

### 5. Связь с другими модулями

| Модуль | Связь |
|--------|--------|
| CORE-01 | Помещения должны существовать в `premises` |
| LOST-02 | Общая страница Upload; блок только для суперадмина |
| ADM-06, ADM-07 | Параллельный импорт контактов (доступен всем админам) — не пересекается |
| BE-03 | Аудит импорта |
| CORE-04 | `area_participated`, `participation_ratio` в API кворума и блок «Статистика ОСС» (SR-CORE04-008) |
| FE-06 | Раскраска шахматки и статистика подъезда по `oss_participation` |

**Вне scope:** шаблон XLSX для выгрузки (аналог ADM-08) не создаётся
