## OPS-01 — Backups: ежедневные дампы (глубина 14 дней) — **РЕАЛИЗОВАНО**

**Реализация:** скрипт вне репозитория (напр. `~/mkd-contacts-backup/`): `daily-backup.sh` — получение дампа по SSH с хоста приложения (обёртка или `docker compose exec -T db pg_dump`), манифест, копирование в каталоги на хосте, ротация (KEEP_COUNT). Cron 2× в сутки (1:00 и 13:00 МСК). Деплой в репо: [scripts/deploy-release.sh](../../scripts/deploy-release.sh) перед обновлением кода создаёт дамп в `backups/` (без ротации).

### Вариант реализации: сохранение на этот хост

Хранение бэкапов на том же сервере, где запущено приложение. SSH на хост уже настроен (для деплоя и администрирования). S3 при необходимости можно добавить позже как второй этап.

### План деталей OPS-01 (на этот хост)

| Элемент | Решение |
|--------|---------|
| **Где запускать** | Cron на хосте (пользователь, под которым крутится приложение / деплой). |
| **Что делать** | Раз в сутки: `pg_dump` из контейнера `db` → файл в каталог на хосте. |
| **Каталог на хосте** | `$APP_DIR/backups/` (как при деплое) или отдельный, например `/var/backups/mkd-contacts/` (нужны права на запись). |
| **Имена файлов** | `backup_YYYYMMDD_HHMMSS.sql` + `backup_YYYYMMDD_HHMMSS.manifest.txt` (как в deploy-release.sh) — по одному дампу в сутки достаточно, время можно фиксировать (например 03:00). |
| **Ротация** | Хранить 14 последних дампов; остальные удалять (по дате в имени файла или по mtime). |
| **Скрипт** | Новый `scripts/daily-backup.sh`: читает `POSTGRES_USER`, `POSTGRES_DB` из `.env`, вызывает `docker compose exec -T db pg_dump ...`, пишет манифест (дата, git rev), удаляет бэкапы старше 14 дней. Запуск из `APP_DIR`. |
| **Cron** | Одна строка, например `0 3 * * * cd /path/to/mkd-contacts && ./scripts/daily-backup.sh >> /var/log/mkd-daily-backup.log 2>&1`. |
| **Логирование** | Скрипт пишет в stdout/stderr (успех/ошибка); cron перенаправляет в лог. При ошибке pg_dump — ненулевой exit code, можно при желании подключить OPS-03 (алерт). |
| **Безопасность** | Каталог backups/ не в git (.gitignore уже есть). Права на каталог — только владелец/группа, запускающая cron. |

Открытые решения перед реализацией:
- Точный путь к каталогу бэкапов на хосте: `$APP_DIR/backups` или отдельный `/var/backups/mkd-contacts`.
- Время запуска cron (например 03:00 по Europe/Moscow).
- Нужна ли отправка копии на другой сервер по SSH (scp/rsync) — можно добавить опциональным шагом после сохранения на хост.

### 1. Архитектурный анализ (CoT)
Реализуется вне основного приложения: скрипт + cron на хосте. Дамп сохраняется локально; при необходимости позже — второй этап: копия в S3 или на другой сервер по SSH.

### 2. Описание и цель
Ежедневное резервное копирование БД с ротацией 14 дней. Цель — БТ и RTO. Первый этап: сохранение на этот хост.

### 3. Функциональные требования
* **SR-OPS01-001:** Система (скрипт/задача) должна выполнять создание дампа БД (полный или по выбранной схеме) не реже одного раза в сутки.
* **SR-OPS01-002:** Дамп должен сохраняться в каталог на хосте, где запущено приложение (или в указанный в конфигурации каталог).
* **SR-OPS01-003:** Должна сохраняться глубина не менее 14 копий (14 дней при ежедневном запуске) с удалением более старых.
* **SR-OPS01-004:** Учётные данные БД (для pg_dump) берутся из .env или окружения; секреты не коммитить.
* **SR-OPS01-005:** При сбое создания дампа система должна логировать ошибку и при необходимости уведомлять ответственных (см. OPS-03).

### 4. Сценарий использования
**Триггер:** По расписанию (ежедневно, cron).
**Main Success Scenario:**
1. Cron запускает скрипт; скрипт создаёт дамп PostgreSQL в каталог на хосте, пишет манифест, удаляет дампы старше 14 дней.
2. Лог фиксирует успех.
**Alternative Flows:**
* **AF-1 (Недоступность БД):** Дамп не создаётся; логировать; не удалять последнюю успешную копию.
* **AF-2 (Нет места на диске):** Логировать; при необходимости очистить старые дампы вручную или уменьшить глубину.

### 5. Модель данных
Не применимо.

### 6. Интеграции и API
Локально: вызов `docker compose exec -T db pg_dump ...`. Опционально позже: S3 или scp/rsync на другой хост по SSH.

### 7. Нефункциональные требования
* **Security:** Шифрование дампа при хранении (S3 server-side или клиентское) и ограничение доступа по IAM/политикам.
* **Compliance:** Хранилище в РФ при требовании локализации данных.
