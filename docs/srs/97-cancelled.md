# Отменённые фичи SRS

Фичи, от решения по которым отказались. Текст сохранён для истории требований.

## FE-01 — SVG-навигация: 3-уровневый компонент (id=cadastral_number) — **ОТМЕНЕНА**

> **Решение:** фича не реализуется. Выбор помещения остаётся через каскадные фильтры (FE-03) и шахматку контактов (FE-06, [13-home-premises-chessboard.md](13-home-premises-chessboard.md)).

### 1. Архитектурный анализ (CoT)
Реализуется на Frontend (React): три уровня — мастер-план (подъезды/паркинг), схема разреза (этажи), план этажа (квартиры/мм/кладовки). Элементы SVG содержат id, соответствующий кадастровому номеру или идентификатору помещения в БД. Данные для подсветки состояний запрашиваются с API (CORE-04, контакты по помещениям).

### 2. Описание и цель
Интерактивная SVG-карта для выбора помещения: мастер-план → схема разреза → план этажа. Цель — наглядный выбор помещения с привязкой к кадастровым номерам.

### 3. Функциональные требования
* **SR-FE01-001:** Компонент должен отображать первый уровень (мастер-план): выбор подъезда или паркинга в виде векторной графики (SVG).
* **SR-FE01-002:** После выбора подъезда/зоны компонент должен отображать второй уровень (схема разреза): выбор этажа.
* **SR-FE01-003:** После выбора этажа компонент должен отображать третий уровень (план этажа): выбор помещения (квартира/машиноместо/кладовка) с id, соответствующим кадастровому номеру или premise_id в БД.
* **SR-FE01-004:** Каждый кликабельный объект SVG должен иметь атрибут id, соответствующий идентификатору помещения в системе (кадастровый номер или внутренний id).
* **SR-FE01-005:** При выборе помещения система должна устанавливать текущее выбранное помещение для последующей передачи в форму (FE-04).
* **SR-FE01-006:** Компонент должен поддерживать pinch-to-zoom и pan для паркинга и кладовок (по БТ).

### 4. Сценарий использования
**Триггер:** Пользователь открывает страницу с картой объекта.
**Main Success Scenario:**
1. Пользователь выбирает подъезд → этаж → помещение на плане.
2. Система фиксирует выбор и даёт возможность перейти к форме анкеты.
**Alternative Flows:**
* **AF-1 (Нет SVG для объекта):** Отображать сообщение «Карта недоступна» и ссылку на каскадные фильтры (FE-03).
* **AF-2 (Медленная загрузка):** Показывать скелетон/лоадер до появления SVG.

### 5. Модель данных
Не меняется; используются premise_id / cadastral_number из реестра. SVG-файлы или конфигурация слоёв должны содержать маппинг id элементов к этим идентификаторам.

### 6. Интеграции и API
Клиент запрашивает метаданные карты и состояния помещений (для легенды FE-02), например GET /api/buildings/{id}/premises/state или данные в составе ответа по объекту.

### 7. Нефункциональные требования
* **Accessibility:** Текстовая альтернатива — каскадные фильтры (FE-03). SVG с aria-label где применимо.
* **Performance:** Ленивая загрузка уровней; не грузить все этажи сразу при открытии мастер-плана.

## CORE-06 — Математика кворума: сложный расчёт долей (API) — **ОТМЕНЕНА**

> **Решение:** фича не реализуется. Используется базовый расчёт кворума (CORE-04, [08-core04-quorum.md](08-core04-quorum.md)).

### 1. Архитектурный анализ (CoT)
Расширение Logic Module (CORE-04): расчёт доли площади, где *все* валидированные собственники по помещению проголосовали «ЗА», к общей площади дома. Учитываются только помещения с полной валидацией и единогласно «ЗА». Данные из той же БД.

### 2. Описание и цель
Сложный вариант прогноза кворума: в знаменатель идут только площади помещений, по которым все валидированные контакты — «ЗА». Цель — консервативная оценка по БТ.

### 3. Функциональные требования
* **SR-CORE06-001:** Система должна определять помещения, по которым все активные (не удалённые, не неактуальные) валидированные контакты имеют в связанной записи голосование_в_ОСС позиция_за = true.
* **SR-CORE06-002:** Система должна суммировать площади только таких помещений.
* **SR-CORE06-003:** Система должна вычислять отношение этой суммы к общей площади МКД и возвращать долю и флаг достижения порога 2/3.
* **SR-CORE06-004:** Система должна предоставлять API для получения этого расчёта по идентификатору дома (отдельный эндпоинт или параметр mode=strict в CORE-04).
* **SR-CORE06-005:** Помещения без валидированных контактов или с хотя бы одним «против» не входят в сумму для сложного расчёта.

### 4. Сценарий использования
**Триггер:** Запрос отчёта или страницы аналитики с «строгим» расчётом кворума.
**Main Success Scenario:**
1. Клиент запрашивает сложный расчёт по building_id.
2. Сервер считает площадь «все валидированы и ЗА», возвращает ratio и quorum_reached.
**Alternative Flows:**
* **AF-1 (Нет помещений с полным согласием):** Возврат ratio = 0, quorum_reached = false.
* **AF-2 (Делитель ноль):** Аналогично CORE-04 — избегать деления на ноль.

### 5. Модель данных
Используются помещения, контакты (status), голосование_в_ОСС (позиция_за). Логика: по каждому помещению проверить «все валидированные контакты имеют позиция_за = true».

### 6. Интеграции и API
* **Method:** `GET /api/buildings/{building_id}/quorum?mode=strict`
* **Response:** Тот же формат, что в CORE-04; значения считаются по правилам «только единогласно валидированные ЗА».

### 7. Нефункциональные требования
* **Performance:** Возможна более тяжёлая агрегация; индексы и при необходимости кеш с коротким TTL.
