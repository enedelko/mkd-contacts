## FE-03 — Базовый поиск: каскадные фильтры

### 1. Архитектурный анализ (CoT)
Реализуется на Frontend (React): последовательный выбор Подъезд → Этаж → Тип помещения → Номер. Данные для вариантов фильтров запрашиваются с Backend из PostgreSQL (на основе реестра CORE-01). Служит текстовой альтернативой SVG-карте.

### 2. Описание и цель
Каскадный поиск помещения: выбор из списков «Подъезд» → «Этаж» → «Тип помещения» → «Номер». Цель — дать пользователю возможность выбрать помещение без карты (слабые устройства, доступность).

### 3. Функциональные требования
* **SR-FE03-001:** Система должна предоставлять API для получения списка уникальных подъездов по объекту (МКД/кадастр дома).
* **SR-FE03-002:** Система должна предоставлять API для получения списка этажей по выбранному подъезду.
* **SR-FE03-003:** Система должна предоставлять API для получения списка типов помещений по подъезду и этажу.
* **SR-FE03-004:** Система должна предоставлять API для получения списка номеров помещений по подъезду, этажу и типу.
* **SR-FE03-005:** После выбора номера система должна однозначно определять помещение (идентификатор для последующей формы FE-04).
* **SR-FE03-006:** Клиент должен отображать каскадные выпадающие списки (или эквивалент) и обновлять следующие уровни при смене предыдущего.
* **SR-FE03-008:** Списки в каскадных фильтрах должны быть отсортированы в натуральном (числовом) порядке: этажи — по числовому значению (−2, −1, 1, 2, … 10, 11, …); номера помещений — по числовой части нормализованного номера, затем по буквенному суффиксу (1, 2, 5a, 5b, 9, 10, 12a). Для сортировки номеров помещений используется функция нормализации (SR-FE03-007).
* **SR-FE03-007:** Каскадные фильтры и поиск по номеру помещения должны использовать **ту же функцию нормализации**, что и импорт (CORE-01): приведение к нижнему регистру и удаление пробелов, очистка от префиксов (кв, пом, оф, №, подвал), мапинг кириллических литер («а»/«б»/«в» → латиница), преобразование римских чисел (I–XXX) в арабские. Пользовательский ввод перед запросом к API автоматически преобразуется в нормализованный `room_id` (например, «IX» → «9», «5-Б» → «5b», «кв. IX-б» → «9b»).

### 4. Сценарий использования
**Триггер:** Пользователь открывает страницу поиска помещения (без карты).
**Main Success Scenario:**
1. Пользователь выбирает подъезд → этаж → тип → номер.
2. Клиент запрашивает с сервера варианты для каждого уровня.
3. После выбора номера отображается выбранное помещение и возможность перейти к форме (FE-04).
**Alternative Flows:**
* **AF-1 (Нет данных):** Если реестр не загружен, API возвращает пустые списки; клиент отображает сообщение «Данные по дому пока не загружены».
* **AF-2 (Таймаут):** При таймауте запроса клиент отображает сообщение об ошибке и предлагает повторить.

### 5. Модель данных
Модель помещений — по LOST-01 ([03-basic-admin.md](03-basic-admin.md)); данные наполняются импортом CORE-01. Для каскадных фильтров используются атрибуты: подъезд, этаж, тип_помещения, номер_помещения (String), идентификатор помещения (кадастровый_номер или id). Иерархия подъезд → этаж → тип → номер совпадает с CORE-01/LOST-01.

### 6. Интеграции и API
* **Method:** `GET /api/premises/filters?building_id={id}&entrance={}&floor={}&type={}`  
  Параметр `building_id` задаёт объект (МКД/дом); при единственном доме в реестре может быть константой или опускаться (см. реализацию и LOST-01).
* **Response (список подъездов):**
```json
{
  "entrances": ["1", "2", "3"]
}
```
* **Response (этажи):**
```json
{
  "floors": [1, 2, 3, 4, 5]
}
```
* **Response (типы):**
```json
{
  "types": ["квартира", "машиноместо", "кладовка"]
}
```
* **Response (номера и итоговый premise_id):** Идентификатор помещения — кадастровый_номер (см. LOST-01).
```json
{
  "premises": [
    { "number": "45", "premise_id": "кадастровый_номер" }
  ]
}
```

### 7. Нефункциональные требования
* **Performance:** Ответы списков — кешируемые на уровне HTTP (короткий Cache-Control) или быстрые запросы к БД с индексами по подъезд/этаж/тип.

## FE-04 — Форма сбора данных (UI)

### 1. Архитектурный анализ (CoT)
Форма реализуется на Client (React SPA): поля ввода, отправка на Backend. Backend (Logic Module) принимает данные, проверяет лимиты и капчу, сохраняет в БД. Требуется интеграция с капчей (Turnstile) и блоком согласий по 152-ФЗ.

### 2. Описание и цель
UI-форма для сбора контактов и позиции по ОСС: переключатель статуса (собственник / проживающий), контакт (телефон, Telegram, email), вопросы ОСС (одобрение схемы шлагбаумов, формат голосования), обязательные согласия на ОПД и ссылка на Политику. Все формулировки — гендерно-нейтральные. Цель — сбор данных в соответствии с БТ и минимизацией ПДн.

### 3. Функциональные требования
* **SR-FE04-001:** Форма должна содержать поле выбора помещения (читаемое отображение выбранного помещения после выбора по карте/фильтрам).
* **SR-FE04-002:** Форма должна содержать переключатель (radio) статуса: «Собственник помещения» / «Проживающий (не собственник)». По умолчанию выбран «Собственник помещения».
* **SR-FE04-003:** Форма должна содержать поля контакта: хотя бы одно из полей «Телефон», «Telegram», «Email» обязательно для заполнения, **кроме** случая анонимного голосования собственника (SR-FE04-014). 
* **SR-FE04-004:** Форма должна содержать вопрос «Одобряю схему размещения шлагбаумов» с вариантами (radio): «За» / «Против» / «Ещё не определились» (поле `barrier_vote`: `for` | `against` | `undecided`).
* **SR-FE04-005:** Форма должна содержать вопрос «Как вы проголосуете» с вариантами (radio): «Онлайн в Электронном доме» / «На бумажном бюллетене» / «Ещё не определились» (поле `vote_format`: `electronic` | `paper` | `undecided`).
* **SR-FE04-006:** Форма должна содержать вопрос о статусе в Электронном доме с вариантами (radio): «Не пользуюсь ЭД / нет данных», «ЭД установлен, требуется подтвердить собственность», «Собственность подтверждена и видна» (поле `registered_ed`: `none` | `account` | `owner`) и ссылкой на https://ed.mos.ru/about-oss/.
* **SR-FE04-007:** Форма должна содержать чекбокс согласия на обработку ПДн и ссылку на Политику конфиденциальности (версия `1.0`). Чекбокс обязателен при заполнении контактных данных. Чекбокс и ссылка размещены внутри блока контактов и отображаются только при наличии заполненного контакта. При анонимной отправке (без контактов) чекбокс не требуется; передаётся `consent_version: "IP"` — согласие фиксируется по IP-адресу. Политика размещена на отдельной странице `/policy` в составе приложения.
* **SR-FE04-013:** На странице Политики конфиденциальности в разделе 9 «Контактная информация» система должна отображать актуальный перечень администраторов Приложения (ФИО и помещение), получаемый через публичный API. Перечень включает всех администраторов и суперадминистраторов.
* **SR-FE04-008:** Форма должна инициировать отображение виджета капчи (Turnstile) перед отправкой.
* **SR-FE04-009:** Клиент должен отправлять данные на сервер только после успешного прохождения капчи и валидации полей на стороне клиента.
* **SR-FE04-010:** При успешной отправке форма должна показывать сообщение об успехе; при ошибке — сообщение об ошибке без раскрытия лишних деталей.
* **SR-FE04-011:** При ручном вводе или поиске по номеру помещения клиент должен применять ту же функцию нормализации, что и при импорте (CORE-01) и в FE-03. Пользовательский ввод автоматически преобразуется в нормализованный `room_id` перед запросом к API (например, «IX» → «9», «5-Б» → «5b», «кв. IX-б» → «9b»).
* **SR-FE04-012:** Если пользователь выбрал статус «Проживающий (не собственник)», блок вопросов ОСС (barrier_vote, vote_format, registered_ed) должен скрываться, а соответствующие поля — передаваться на сервер как `null`. При переключении обратно на «Собственник» блок должен появляться снова. Логика применяется как в публичной форме (FE-04), так и в админской форме (ADM-03).
* **SR-FE04-014:** Система должна поддерживать **анонимное голосование по ОСС**: собственник может отправить анкету без контактных данных (phone, email, telegram_id = NULL), если заполнен хотя бы один из вопросов ОСС (barrier_vote, vote_format, registered_ed). Ограничения: (a) не более 1 анонимного контакта на помещение (код ошибки `ANON_VOTE_EXISTS`, HTTP 409); (b) для не-собственников анонимная отправка запрещена — контактные данные обязательны; (c) дедупликация по blind index не применяется (все индексы NULL); (d) при анонимной отправке `consent_version` = `"IP"`, при наличии контактов — `"1.0"`; (e) анонимные голоса учитываются в расчёте кворума наравне с идентифицированными контактами. Риски: невозможность верификации личности, невозможность обратной связи, потенциальное злоупотребление (смягчается капчей, rate limit по IP и лимитом 1 анон/помещение).

### 4. Сценарий использования
**Триггер:** Пользователь выбрал помещение и открыл форму.
**Main Success Scenario:**
1. Пользователь заполняет поля, отмечает согласия, проходит капчу и нажимает «Отправить».
2. Клиент валидирует данные и отправляет запрос на API.
3. Сервер проверяет лимиты и капчу, сохраняет анкету и возвращает успех.
4. Пользователь видит подтверждение.
**Alternative Flows:**
* **AF-1 (Ошибка валидации):** При невалидных полях или непройденной капче клиент не отправляет запрос и отображает ошибки у полей/капчи.
* **AF-2 (Лимит 10 записей/час):** Сервер возвращает 429; клиент отображает сообщение о превышении лимита и возможности повторить позже.

### 5. Модель данных
**Затрагиваемые сущности:** Структура контактов и голосование_в_ОСС — по LOST-01 ([03-basic-admin.md](03-basic-admin.md)). Форма и API submit опираются на: контакт (premise_id, is_owner, phone, email, telegram_id, registered_in_ed, согласия) и запись голосования (barrier_vote — позиция по шлагбаумам, vote_format — формат голосования; поле проголосовал задаётся бэкендом/процессом). Остальные поля контакта (даты, ip, статус) задаёт бэкенд.

### 6. Интеграции и API
* **Method:** `POST /api/submit`
* **Request:** Поля `barrier_vote`, `vote_format` → запись в голосование_в_ОСС; `registered_ed` → контакт.
```json
{
  "premise_id": "cadastral",
  "is_owner": true,
  "phone": "+7...",
  "email": "user@example.com",
  "telegram_id": null,
  "barrier_vote": "for",
  "vote_format": "electronic",
  "registered_ed": "yes",
  "consent_version": "1.0",
  "captcha_token": "turnstile_response_token"
}
```
* **Response (успех):**
```json
{
  "success": true,
  "message": "Данные приняты"
}
```
* **Response (ошибка лимита):**
```json
{
  "detail": "Rate limit exceeded",
  "retry_after": 3600
}
```

### 7. Нефункциональные требования
* **Security:** Передача только по HTTPS; контакты на сервере сохраняются в зашифрованном виде (см. BE-02). Капча обязательна. В тестовой среде допускаются исключения (например, HTTP, отключение капчи) по конфигурации.
* **Logging:** Не логировать значения полей с ПДн в открытом виде; логировать только факт отправки и premise_id.

## ADM-03 — Прямой ввод: форма добавления контактов админом (авто-валидация)

### 1. Архитектурный анализ (CoT)
Реализуется в админ-интерфейсе (React) и Logic Module: форма ввода контакта по выбранному помещению с автоматической установкой статуса «валидирован» (т.к. ввод от доверенного оператора). Используются те же валидации форматов (CORE-02), шифрование (BE-02), лимиты на помещение при необходимости не применяются или ослаблены для админа.

### 2. Описание и цель
Возможность администратора вручную добавить контакт по помещению с немедленной валидацией. Цель — ввод данных из бумажных анкет или по телефону.

### 3. Функциональные требования
* **SR-ADM03-001:** Система должна предоставлять форму (UI) и API для создания записи контакта от имени администратора с привязкой к помещению.
* **SR-ADM03-002:** При создании контакта через админ-форму система должна автоматически устанавливать статус «валидирован».
* **SR-ADM03-003:** Система должна применять валидацию форматов полей (телефон, email, telegram) по правилам CORE-02. 
* **SR-ADM03-004:** Система должна сохранять контакты в зашифрованном виде (BE-02) и записывать в аудит факт создания и идентификатор админа.
* **SR-ADM03-005:** Система не должна требовать капчу для запроса от администратора; проверка JWT достаточна.
* **SR-ADM03-006:** Админ-форма должна содержать переключатель «Собственник помещения» / «Проживающий (не собственник)». При выборе «Проживающий» блок вопросов ОСС скрывается, поля передаются как `null` (см. SR-FE04-012).
* **SR-ADM03-007:** Система должна предоставлять API (`GET /api/admin/contacts/{id}`, `PUT /api/admin/contacts/{id}`) и UI для редактирования существующего контакта. При редактировании помещение не изменяется; обновляются контактные данные, статус собственника и позиция по ОСС. Применяются те же правила валидации (CORE-02) и шифрования (BE-02), что и при создании.

### 4. Сценарий использования
**Триггер:** Администратор выбирает «Добавить контакт» и выбирает помещение.
**Main Success Scenario:**
1. Администратор заполняет поля контакта и позиции по ОСС, отправляет форму.
2. Сервер проверяет права и форматы, шифрует, сохраняет со статусом «валидирован», возвращает успех.
**Alternative Flows:**
* **AF-1 (Ошибка валидации):** Возврат 400 с указанием поля.
* **AF-2 (Помещение не найдено):** Возврат 404.

### 5. Модель данных
Та же сущность контакта; источник/флаг admin_created для аудита.

### 6. Интеграции и API
* **Method:** `POST /api/admin/contacts`
* **Request:**
```json
{
  "premise_id": "77:01:0001001:123",
  "phone": "+7...",
  "email": null,
  "telegram_id": null,
  "barrier_vote": "for",
  "vote_format": "paper",
  "registered_ed": "yes"
}
```
* **Response:** 201 Created с contact_id и статусом validated.

### 7. Нефункциональные требования
* **Security:** Только администратор; аудит создания с user_id.
* **Compliance:** Согласие на ОПД при ручном вводе — 🤔 допущение: фиксировать в записи «согласие получено при личном контакте» или требовать чекбокс в форме.
