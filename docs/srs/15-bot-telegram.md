# BOT-01…04 — Telegram-бот: поиск, анкета, ЛК, синхронизация

> Извлечено из [98-backlog.md](98-backlog.md). Зависимости: CORE-01, CORE-02, BE-02, BE-03.

**Принятые решения:**
- Бот работает с одним МКД (выбор дома не нужен).
- Один бот для логина администраторов (Login Widget) и опроса собственников (webhook). Используется тот же `TELEGRAM_BOT_TOKEN`.
- Поток строится вокруг собственника: сначала все помещения, потом вопросы (один раз), в конце — телефон.
- Анонимное голосование в боте не поддерживается; telegram_id заполняется автоматически из `message.from.id`.
- telegram_id — ПДн. Хранится только в зашифрованном виде (`telegram_id`, Fernet) + Blind Index (`telegram_id_idx`, HMAC-SHA256). Отдельного поля `telegram_user_id` в открытом виде нет.
- Телефон не обязателен — telegram_id достаточен для идентификации.
- Email в боте не собирается.
- Данные сохраняются инкрементально (после каждого ответа). Прерывание на любом шаге не теряет ранее сохранённые данные.
- Rate limit: 10 submit/час на telegram_id_idx (аналог BE-04 по IP, SR-BE04-006); плюс лимит 10 невалидированных на помещение (SR-CORE02-004).
- Удаление помещения: REMOVE_PREMISE → CONFIRM_REMOVE_PREMISE → `status='inactive'`, обнуление ПДн одного контакта, запись в аудит (`action: premise_removed`).
- Удаление телефона: CONFIRM_DELETE → PATCH phone=null. Остальные данные сохраняются.
- Полное забвение: CONFIRM_FORGET → `status='inactive'` для всех контактов, обнуление ПДн и Blind Index, запись в аудит (`action: forget`).
- Поле `deleted_at` не используется. Различие между удалением помещения и забвением — в `action` аудит-лога. Существующий фильтр `WHERE status IN ('pending','validated')` исключает неактивные записи.
- Словарь синонимов типов помещений хранится в БД (таблица `premise_type_aliases`) и загружается при старте. Позволяет добавлять синонимы через админку без передеплоя.
- Нераспознанные вводы пользователей логируются в таблицу `bot_unrecognized` для систематического улучшения алгоритмов. Доступны суперадмину.
- **Рассылка:** суперадмин может отправить уведомление всем пользователям бота (кто хотя бы раз нажимал /start). Список получателей хранится в SQLite бота (таблица `broadcast_recipients`), пополняется при каждом /start.
- **Написать админам:** любой пользователь бота может отправить сообщение всем администраторам системы; бот получает список telegram_id админов через Backend (`GET /api/bot/admins-telegram-ids`) и пересылает текст каждому с пометкой «от пользователя …».
- Автообновление шахматки/карты на фронтенде не требуется; данные из бота видны при перезагрузке страницы.
- Сессии бота персистентны (SQLite-файл на volume) — переживают рестарт контейнера.
- Webhook (не long polling) — HTTPS уже есть через Nginx.

**Реальные типы помещений в БД** (из реестра, 5 типов, 1815 помещений):

| premises_type (в БД) | Кол-во | Номера | Сокращение (short_name) |
|---|---|---|---|
| Квартира | 805 | 1..99 | Кв. |
| Машино-место | 815 | 1..99 | ММ |
| Офис (апартаменты) | 127 | 1..99+ | Ап. |
| Вспомогательное помещение офисов (кладовка) | 42 | 01a..24b | Клад. |
| БКТ | 26 | 1..9 | БКТ |

Номера 1–99 пересекаются у 4 типов. Ввод голого числа (например, «45») даёт 3–4 совпадения → обязателен DISAMBIGUATE с кнопками `[Кв. 45]` `[ММ 45]` `[Ап. 45]`.

## Архитектура

```
┌──────────────┐                                             ┌─────────────────────────────────┐
│  Telegram    │          ┌──────────────┐                   │        Docker Compose           │
│  Cloud       │          │ Nginx (хост) │                   │                                 │
│  (webhook)   │────────▶│ /api/tg-wh/* │──▶ bot:8443 ───▶ │  bot  (aiogram 3.x)             │
│              │          │              │                   │    ├─ FSM (стейт-машина)        │
│              │          │              │                   │    └─ SQLite: /data/sessions.db │
│              │          │              │                   │       (volume: bot_data)        │
│              │          │              │                   │          │                      │
│              │          │              │                   │          │   HTTP               │
│              │          │              │                   │          ▼                      │
│              │          │ /api/*       │──▶ backend:8000  │  backend  (FastAPI, :8000)      │
│              │          │              │                   │          │                      │
│              │          │              │                   │          ▼                      │
│              │          │              │                   │  db      (PostgreSQL)           │
│              │          │              │                   │                                 │
│              │          │ /*           │──▶ frontend:8080 │  frontend (Nginx)               │
│              │          │              │                   └─────────────────────────────────┘ 
└──────────────┘          └──────────────┘
```

**Подход: Бот → Backend API (тонкий клиент).** Бот — отдельный контейнер в docker-compose, обращается к `backend` по внутренней сети Docker. Вся бизнес-логика (валидация, шифрование, аудит, лимиты) — в Backend. Бот отвечает только за Telegram UI и стейт-машину диалога.

**Авторизация бота к Backend:** заголовок `X-Bot-Token` (shared secret, хранится в `.env`, не в репозитории). Backend проверяет токен на эндпоинтах `/api/bot/*`.

**Персистенция сессий:** aiogram 3.x поддерживает кастомные storage для FSM. Используем SQLite-файл (`/data/sessions.db`), примонтированный через Docker volume `bot_data`. При рестарте контейнера незавершённые диалоги сохраняются — пользователь продолжает с того же шага. TTL сессии — 1 час; по истечении бот предлагает начать заново.

## Стейт-машина и CX

Состояние диалога хранится в **SQLite** (`/data/sessions.db`, Docker volume `bot_data`). TTL сессии — 1 час. Каждое состояние хранит: `chat_id`, `state` (enum), `data` (JSON с накопленными ответами), `updated_at`. При `updated_at` старше 1 часа — очистка и переход в IDLE. В той же БД — таблица **broadcast_recipients** (chat_id): список получателей рассылки суперадмином; пополняется при каждом /start.

Поток строится вокруг собственника, а не отдельного помещения: сначала собираем все помещения, потом вопросы (один раз на все помещения), в конце — телефон и политика. Вопросы и телефон показывают текущие ответы, если пользователь уже обращался ранее.

```
IDLE → PREMISES_OVERVIEW → ENTER_PREMISE → [DISAMBIGUATE] → CONFIRM_PREMISE
     → OFFER_PARKING_STORAGE / OFFER_MORE → [ENTER_PARKING_INPUT] → CONFIRM_PREMISE / назад
     → VOTE_METHOD → BARRIER_VOTE
     → CONTACT_MANAGE → [CONTACT_CONSENT] → ENTER_PHONE или назад в CONTACT_MANAGE
                      → [ENTER_PHONE] (если телефон уже был)
                      → [CONFIRM_DELETE → CONTACT_MANAGE]

PREMISES_OVERVIEW / OFFER_PARKING_STORAGE / OFFER_MORE
     → REMOVE_PREMISE → CONFIRM_REMOVE_PREMISE → PREMISES_OVERVIEW

IDLE → MY_DATA_VIEW → [CONFIRM_FORGET → IDLE]
                    → [Изменить → PREMISES_OVERVIEW]

IDLE → BROADCAST_WAIT_TEXT (только суперадмин) → [текст] → рассылка всем → IDLE
IDLE → TO_ADMINS_WAIT_TEXT → [текст] → доставка всем админам → IDLE
```

### Команды бота

| Команда | Действие |
|---------|----------|
| `/start` | Начало: приветствие + [Я собственник] |
| `/mydata` | Просмотр данных: MY_DATA_VIEW (редактирование, забвение) |
| `/cancel` | Сброс текущего диалога → IDLE |
| `/help` | Справка: инструкция по установке и настройке «Электронный дом Москва» (вход через mos.ru, проверка и подтверждение собственности — вкладка «Мой дом», «Настроить» при необходимости, поддержка приложения) и список команд |

* **IDLE**

Приветствие (ниже) может дополняться актуальной сводкой по дому: доля площади в Электронном Доме (%), достигнут ли кворум (2/3), фраза «ОСС стартуем после того, как будет набираться кворум». Данные запрашиваются с `GET /api/buildings/default/quorum`; при недоступности backend приветствие показывается без блока.

```
Этот бот помогает оценить голоса и собрать контакты собственников в ЖК Silver.

Нам важно понять, набирается ли кворум — 2/3 голосов
от общей площади дома.

Если вы собственник помещения в этом доме — нажмите кнопку ниже.
```

### Примечания к CX

1. **Каждый шаг — одно сообщение с InlineKeyboard.** Бот редактирует то же сообщение (`edit_message_text`), чат не превращается в полотно.
2. **ENTER_PHONE** — единственный шаг с ReplyKeyboard (для кнопки Share Contact). На этом шаге `[Назад]` доступен через команду `/back`.
3. **telegram_id подставляется автоматически** из `message.from.id` — пользователь не видит этого.
4. **Телефон не обязателен.** telegram_id всегда известен и достаточен для идентификации. Пользователь может пропустить шаг с телефоном (`[Пропустить]` в CONTACT_MANAGE). Это осознанное решение: порог входа снижается, а связаться с пользователем можно через Telegram.
5. **Инкрементальное сохранение.** Данные сохраняются на Backend после каждого ответа, если они изменились: CONFIRM_PREMISE создаёт/привязывает контакт, VOTE_METHOD сохраняет формат голосования, BARRIER_VOTE — позицию по шлагбаумам, ENTER_PHONE/CONFIRM_DELETE — телефон. Если пользователь прерывает поток на любом шаге (`[Отмена]`), ранее сохранённые ответы остаются в БД. Это ожидаемое поведение.
6. **Вопросы (VOTE_METHOD, BARRIER_VOTE) и телефон (CONTACT_MANAGE) показывают текущий ответ** — возвращающийся пользователь видит, что отвечал раньше, и может изменить или оставить.
7. **Согласие с Политикой** вынесено в отдельный шаг **CONTACT_CONSENT**. Показывается только когда у пользователя ещё нет телефона и он нажимает «Добавить телефон» или вводит номер на шаге CONTACT_MANAGE. Две кнопки: «Согласен» / «Не согласен». При «Не согласен» — возврат на CONTACT_MANAGE. Текст согласия — форма 1 из приложения к Политике (обработка контактных данных для координации ОСС в ЖК Silver, без IP и без «защиты от флуда»). На шаге CONTACT_MANAGE при уже сохранённом телефоне текст согласия не повторяется.
8. **EDOM и формат голосования объединены** в один шаг VOTE_METHOD. Три варианта по ЭД: «Планирую установить ЭД» (registered_in_ed=none), «ЭД установлен, требуется подтвердить собственность» (registered_in_ed=account), «ЭД установлено, собственность видна» (registered_in_ed=owner). Вариант «account» дополнительно подсвечивает пользователю шаг 5 инструкции (подтверждение собственности для участия в ОСС).
9. **barrier_vote и vote_format** в модели данных хранятся per-contact (per-premise). При сохранении бот дублирует ответы во все записи контактов по помещениям собственника.
10. **Три уровня управления данными:** (a) CONFIRM_DELETE — удаляет только телефон; (b) REMOVE_PREMISE → CONFIRM_REMOVE_PREMISE — `status='inactive'` одного контакта, обнуление ПДн, аудит `action: premise_removed`; (c) CONFIRM_FORGET — `status='inactive'` всех контактов, обнуление ПДн + Blind Index, аудит `action: forget`. Поле `deleted_at` не используется — различие через `action` в аудит-логе.
11. **Короткие названия типов в кнопках.** Бот использует `short_name` из таблицы `premise_type_aliases`: «Кв. 45», «ММ 12», «Ап. 45», «Клад. 5a», «БКТ 3». Полное название показывается в CONFIRM_PREMISE.
12. **Нераспознанные вводы** логируются в `bot_unrecognized` (input_text + telegram_id_idx + timestamp). Суперадмин видит частотные паттерны и добавляет синонимы через страницу «Словарь бота».
13. **Всего 20 состояний** (в т.ч. CONTACT_CONSENT). `/cancel` из любого состояния ведёт в IDLE.
14. **Рассылка (суперадмин):** кнопка [Рассылка] в IDLE видна только при `GET /api/bot/me/role` → role = super_administrator. Получатели — таблица `broadcast_recipients` в SQLite бота.
15. **Написать админам:** кнопка [Написать админам] доступна всем. Список админов — `GET /api/bot/admins-telegram-ids`.

## BOT-01 — Алгоритмы поиска: Fuzzy Search (Левенштейн) и Regex кадастровых номеров

### 1. Архитектурный анализ (CoT)
Реализуется на Backend: нормализация ввода (переиспользуется функция из SR-FE03-007/CORE-01, см. SR-CORE01-018–023), поиск по кадастровому номеру через regex, парсинг типа помещения и номера по словарю, при отсутствии точного совпадения — fallback по нормализованному номеру помещения (без изменения схемы БД), fuzzy matching по Левенштейну. Нормализация номера помещения: в доме допустимы только постфиксы А и Б; кириллическую «в» пользователи часто путают с латинской B — нормализация даёт «в» → «b» (например, 3В → 3b). Заменяются только а, б, в → a, b, b; остальные кириллические литеры не заменяются. Поиск ведётся в рамках единственного МКД (building_id не требуется).

### 2. Описание и цель
Распознавание намерения пользователя в текстовом вводе: извлечение кадастрового номера или «тип + номер» помещения с учётом опечаток. Цель — поиск помещения в боте по БТ [cite: 83, 84, 140–150, 180–181].

### 3. Функциональные требования
* **SR-BOT01-001:** Система должна нормализовать ввод: нижний регистр, удаление лишних пробелов, замена «ё» на «е» (переиспользовать SR-FE03-007).
* **SR-BOT01-002:** Система должна проверять ввод на соответствие шаблону кадастрового номера (regex) и при совпадении выполнять поиск помещения по кадастровому номеру в БД.
* **SR-BOT01-003:** Система должна парсить тип помещения и номер по словарю синонимов. Словарь хранится в БД (таблица `premise_type_aliases`): каждая строка связывает `alias` (синоним, например «кв», «мм», «ап») с `premises_type` (точное значение из `premises`) и `short_name` (сокращение для кнопок, например «Кв.», «ММ»). Загружается при старте, кэшируется. Редактируется суперадмином через веб-интерфейс.
* **SR-BOT01-003a:** Нормализация номера помещения выполняется той же функцией, что и при импорте (CORE-01, `room_normalizer`): допустимы только постфиксы А и Б; «в» трактуется как путаница с B и заменяется на «b» (3В → 3b). Заменяются только а, б, в → a, b, b.
* **SR-BOT01-003b:** Если поиск по точному совпадению `premises_number` и по `LOWER(premises_number)` не дал результатов, система должна выполнить fallback: выбрать кандидатов из БД (по типу помещения при известном типе, иначе все помещения), для каждого вычислить нормализованный номер помещения (`normalize_room_number`) и вернуть те записи, у которых он совпадает с нормализованным вводом пользователя (макс. 5). Схему БД не менять.
* **SR-BOT01-004:** Система должна применять нечёткое сравнение (расстояние Левенштейна, библиотека `rapidfuzz`) для сопоставления ввода со списком помещений при опечатках.
* **SR-BOT01-005:** Система должна возвращать лучшие совпадения (один или несколько, макс. 5) с идентификатором помещения (premise_id), полным названием (display) и сокращённым (short_display, через `short_name` из `premise_type_aliases`).
* **SR-BOT01-006:** При отсутствии совпадений система должна возвращать явный результат «не найдено» и записывать нераспознанный ввод в таблицу `bot_unrecognized` (input_text, telegram_id_idx, created_at) для анализа суперадмином.

### 4. Сценарий использования
**Триггер:** Пользователь бота вводит текст («кв 45», «мместо 5»).
**Main Success Scenario:**
1. Бот передаёт текст на Backend (`POST /api/bot/resolve-premise`).
2. Backend нормализует, проверяет КН, парсит тип/номер, при необходимости fuzzy match.
3. Возвращается список совпадений; бот предлагает выбрать помещение или переходит к анкете.
**Alternative Flows:**
* **AF-1 (Неоднозначный ввод):** Возврат нескольких вариантов; бот показывает InlineKeyboard с вариантами.
* **AF-2 (Некорректный ввод):** Возврат «не найдено»; бот предлагает ввести снова или перейти к каскадному выбору.

### 5. Модель данных
Используется реестр помещений (CORE-01). Словарь синонимов — таблица `premise_type_aliases` (см. Принятые решения). Нераспознанные вводы — таблица `bot_unrecognized`.

### 6. Интеграции и API
* **Method:** `POST /api/bot/resolve-premise` (авторизация: `X-Bot-Token`).
* **Request:**
```json
{
  "text": "кв 45"
}
```
* **Response:**
```json
{
  "matches": [
    { "premise_id": "77:06:...", "display": "Квартира 45", "short_display": "Кв. 45", "confidence": 1.0 }
  ]
}
```

### 7. Нефункциональные требования
* **Performance:** Fuzzy search по всем помещениям дома (одного МКД); лимит на длину ввода (100 символов) и макс. 5 возвращаемых совпадений.

## BOT-02 — Бот-анкета: сбор помещений, вопросы, контакт

### 1. Архитектурный анализ (CoT)
Реализуется в Telegram-боте (webhook): пошаговый FSM-сценарий. Поток строится вокруг собственника: сначала все помещения, затем вопросы (формат голосования, шлагбаумы), в конце — телефон. Данные сохраняются на Backend инкрементально после каждого ответа. telegram_id заполняется автоматически из `message.from.id`. Email не собирается. Телефон не обязателен (telegram_id достаточен для идентификации).

### 2. Описание и цель
Линейный опрос в Telegram для сбора данных собственника: помещения, позиция по голосованию, контакт. Отличия от веб-формы: (1) собираются все помещения собственника за один проход, (2) email не собирается, (3) телефон опционален, (4) вопросы задаются один раз на все помещения. Цель — альтернативный канал по БТ [cite: 80, 81, 92, 93].

### 3. Функциональные требования
* **SR-BOT02-001:** Бот должен вести пользователя по шагам (см. стейт-машину): помещения (одно или несколько, через BOT-01) → формат голосования (ЭД / бумага / не голосую) → позиция по шлагбаумам → телефон (опционально) → сводка.
* **SR-BOT02-002:** Бот должен поддерживать кнопку Share Contact (Telegram ReplyKeyboard) для передачи номера телефона (шаг ENTER_PHONE).
* **SR-BOT02-003:** Согласие на обработку ПДн — отдельный шаг **CONTACT_CONSENT**. Показывается только при первом добавлении телефона (когда телефона ещё нет): при нажатии «Добавить телефон» или при вводе номера на шаге CONTACT_MANAGE. Две кнопки: «Согласен» / «Не согласен». При «Не согласен» — возврат на CONTACT_MANAGE. Текст — форма 1 из приложения к Политике конфиденциальности (контактные данные, координация ОСС в ЖК Silver; без IP и без «защиты от флуда»).
* **SR-BOT02-004:** Данные сохраняются инкрементально: CONFIRM_PREMISE → создание/привязка контакта; VOTE_METHOD → vote_format + registered_in_ed; BARRIER_VOTE → barrier_vote; ENTER_PHONE → phone; при нажатии «Согласен» на CONTACT_CONSENT с сохранённым введённым номером (pending_phone) — PATCH phone сразу, затем переход в CONTACT_MANAGE. Каждый шаг выполняет POST/PATCH на Backend с `source: "telegram"` и `telegram_user_id`.
* **SR-BOT02-005:** При подтверждении помещения типа «квартира/апартаменты» бот предлагает добавить кладовку и/или машиноместо (OFFER_PARKING_STORAGE). После добавления других типов — предлагает добавить ещё (OFFER_MORE).
* **SR-BOT02-006:** Капча для бота не требуется. Rate limit: 10 submit/час по telegram_id (аналог BE-04); плюс лимит 10 невалидированных на помещение (SR-CORE02-004).
* **SR-BOT02-007:** Анонимное голосование в боте не поддерживается; telegram_id всегда сохраняется.
* **SR-BOT02-008:** Телефон не обязателен. telegram_id достаточен для идентификации собственника. Пользователь может пропустить шаг с телефоном.
* **SR-BOT02-009:** Вопросы (VOTE_METHOD, BARRIER_VOTE) и телефон (CONTACT_MANAGE) показывают текущий сохранённый ответ, если пользователь обращался ранее.
* **SR-BOT02-010:** Пользователь может убрать ошибочно добавленное помещение (REMOVE_PREMISE → CONFIRM_REMOVE_PREMISE). Backend: `status='inactive'`, обнуление зашифрованных полей этого контакта, запись в аудит (`action: premise_removed`). Остальные помещения и ответы не затрагиваются.
* **SR-BOT02-011:** Рассылка уведомлений всем пользователям бота доступна только суперадминистратору. Кнопка [Рассылка] в IDLE отображается только при `GET /api/bot/me/role` → role = super_administrator. Получатели — таблица `broadcast_recipients` в SQLite бота (chat_id добавляется при каждом /start). После ввода текста бот отправляет сообщение каждому получателю.
* **SR-BOT02-012:** Отправка сообщения администраторам доступна любому пользователю бота. Кнопка [Написать админам] в IDLE. Бот запрашивает `GET /api/bot/admins-telegram-ids`, затем пересылает введённый текст каждому админу с пометкой об отправителе (username или id).

### 4. Сценарий использования
**Триггер:** Пользователь нажимает `[Я собственник]` или `/start`.
**Main Success Scenario:**
1. Пользователь добавляет помещения (одно или несколько).
2. Отвечает на вопросы о формате голосования и шлагбаумах.
3. Проверяет/добавляет/пропускает телефон.
4. Видит сводку на экране DONE.
Данные сохраняются после каждого шага.
**Alternative Flows:**
* **AF-1 (Отмена на шаге):** `/cancel` или `[Отмена]` — сессия сбрасывается; ранее сохранённые ответы остаются в БД.
* **AF-2 (Лимит на помещение):** Backend возвращает 409; бот сообщает «По этому помещению уже достаточно записей».
* **AF-3 (Rate limit):** Backend возвращает 429; бот сообщает «Слишком много отправок, попробуйте позже».
* **AF-4 (Возвращающийся пользователь):** При наличии существующих данных PREMISES_OVERVIEW показывает список помещений; вопросы показывают предыдущие ответы.
* **AF-5 (Удаление помещения):** Пользователь нажимает `[Убрать помещение]` → выбирает помещение → подтверждает → Backend: `status='inactive'` + обнуление ПДн контакта → возврат в PREMISES_OVERVIEW. Если удалено последнее помещение — PREMISES_OVERVIEW показывает пустое состояние.

### 5. Модель данных
Та же модель контакта (FE-04). Изменения:
- Поле `source` в contacts: varchar(16), default `'web'`, допустимые значения: `web` | `telegram` | `admin`
- telegram_id хранится в существующих полях `telegram_id` (зашифрованный, Fernet) и `telegram_id_idx` (Blind Index, HMAC-SHA256). Отдельного поля `telegram_user_id` в открытом виде нет (ПДн).
- `barrier_vote` и `vote_format` хранятся per-contact (per-premise); при сохранении бот дублирует ответы во все записи контактов по помещениям собственника
- Таблица `premise_type_aliases` — словарь синонимов типов помещений (см. BOT-01)
- Таблица `bot_unrecognized` — лог нераспознанных вводов (см. BOT-01)
- Миграция Alembic

### 6. Интеграции и API

Все эндпоинты: авторизация `X-Bot-Token`.

**Привязка помещения:**
* **Method:** `POST /api/bot/premises`
* **Request:**
```json
{
  "telegram_user_id": "123456789",
  "premise_id": "77:06:..."
}
```
Backend шифрует telegram_user_id (Fernet → `telegram_id`), вычисляет Blind Index (`telegram_id_idx`), создаёт контакт с `source='telegram'`.
* **Response (успех):** 201 Created.
* **Response (лимит):** 409 `{"detail": "Premise limit exceeded"}`.

**Обновление ответов (по всем помещениям собственника):**
* **Method:** `PATCH /api/bot/me/answers`
* **Request:**
```json
{
  "telegram_user_id": "123456789",
  "vote_format": "electronic",
  "registered_in_ed": "owner",
  "barrier_vote": "for",
  "phone": "+79991234567"
}
```
Передаются только изменённые поля. Backend находит все контакты по `telegram_id_idx` (Blind Index) и обновляет. Бот присылает `registered_in_ed` как `"none"` | `"account"` | `"owner"` (расширенный enum). Для обратной совместимости Backend также принимает старые значения `"true"`/`"false"`/`"yes"`/`"no"` и маппит их в `"owner"`/`"none"`.
* **Response:** 200 OK.
* **Response (rate limit):** 429 `{"detail": "Rate limit exceeded", "retry_after": 3600}`.

**Удаление помещения:**
* **Method:** `DELETE /api/bot/me/premises/{premise_id}` (premise_id в path — кадастровый номер; в URL кодировать, т.к. содержит `:`).
* **Request (body — PremiseBody, оба поля обязательны):**
```json
{
  "telegram_user_id": "123456789",
  "premise_id": "77:02:0015008:5364"
}
```
* **Response (успех):** 200 OK `{"detail": "Premise removed"}`.
* **Response (не найдено / чужой):** 404.
* Backend: найти контакт по `premise_id + telegram_id_idx` → `status='inactive'`, обнуление зашифрованных полей и Blind Index → запись в аудит (`action: premise_removed`).

**Получение данных пользователя:**
* **Method:** `GET /api/bot/me/data?telegram_user_id={id}`
Backend находит контакты по `telegram_id_idx` (Blind Index от переданного telegram_user_id).
* **Response:** В БД и в ответе `registered_in_ed` — строка `"none"` | `"account"` | `"owner"` (расширенный enum статуса ЭД). Для обратной совместимости бот при отображении также учитывает старые значения `"yes"`/`"true"` (как `owner`).
```json
{
  "premises": [
    { "premise_id": "77:06:...", "display": "Квартира 45", "short_display": "Кв. 45" },
    { "premise_id": "77:06:...", "display": "Машино-место 12", "short_display": "ММ 12" }
  ],
  "vote_format": "electronic",
  "registered_in_ed": "owner",
  "barrier_vote": "for",
  "phone": "+79991234567"
}
```

**Роль пользователя (для рассылки):**
* **Method:** `GET /api/bot/me/role?telegram_user_id={id}`
* **Response:** `{"role": "super_administrator" | "administrator" | null}`. null — пользователь не в белом списке админов. Используется ботом для отображения кнопки [Рассылка] только суперадмину.

**Список telegram_id админов (для «Написать админам»):**
* **Method:** `GET /api/bot/admins-telegram-ids`
* **Response:** `{"telegram_ids": ["123456789", ...]}` — список всех записей из таблицы admins. Бот пересылает сообщение пользователя каждому.

### 7. Нефункциональные требования
* **Security:** `X-Bot-Token` (shared secret, не Telegram-токен) для авторизации бот→бэкенд. Rate limit 10 submit/час по `telegram_id_idx` (in-memory, аналог BE-04 по IP).
* **Compliance:** Согласие — отдельный шаг CONTACT_CONSENT (форма 1 из приложения к Политике, кнопки «Согласен»/«Не согласен»). consent_version фиксируется при создании контакта (Backend при POST premises уже передаёт 1.1).

## BOT-03 — Просмотр данных, редактирование и «Право на забвение»

### 1. Архитектурный анализ (CoT)
Реализуется в том же боте: команда `/mydata` → состояние MY_DATA_VIEW. Backend находит записи по Blind Index от telegram_user_id и возвращает сводку (помещения, ответы, телефон). Редактирование — переход в основной поток (PREMISES_OVERVIEW) с предзаполнением. Полное удаление («Право на забвение», CONFIRM_FORGET) — soft delete всех контактов: обнуление ПДн включая TG ID, смена статуса, запись в аудит. Удаление только телефона — CONFIRM_DELETE (из CONTACT_MANAGE).

### 2. Описание и цель
Три уровня управления данными: (1) просмотр и редактирование через основной поток, (2) удаление отдельного помещения (`status='inactive'`, обнуление ПДн) или только телефона, (3) полное удаление (забвение — `status='inactive'` для всех контактов, обнуление ПДн и Blind Index). Различие — в `action` аудит-лога. Цель — соответствие БТ [cite: 92, 93] и 152-ФЗ.

### 3. Функциональные требования
* **SR-BOT03-001:** Бот должен предоставлять команду `/mydata` и кнопку `[Мои данные]` (IDLE) для входа в MY_DATA_VIEW.
* **SR-BOT03-002:** В MY_DATA_VIEW бот запрашивает Backend (`GET /api/bot/me/data`) и показывает все данные пользователя: список помещений, ответы на вопросы, телефон.
* **SR-BOT03-003:** Из MY_DATA_VIEW доступны действия: `[Изменить данные]` → переход в PREMISES_OVERVIEW (основной поток с предзаполнением); `[Удалить все мои данные]` → CONFIRM_FORGET.
* **SR-BOT03-004:** При подтверждении полного удаления (CONFIRM_FORGET) Backend: `status='inactive'` для всех контактов данного telegram_user_id, обнуление зашифрованных полей (phone, telegram_id, how_to_address) и Blind Index (`telegram_id_idx`, `phone_idx`). В аудит-лог: `action: forget`, entity_id = blind_index (до обнуления), timestamp. После обнуления Blind Index повторный запрос вернёт пустой результат.
* **SR-BOT03-005:** Удаление только телефона (CONFIRM_DELETE из CONTACT_MANAGE) — PATCH с phone=null. Остальные данные (помещения, ответы) сохраняются.
* **SR-BOT03-006:** Удаление отдельного помещения (REMOVE_PREMISE → CONFIRM_REMOVE_PREMISE) — `DELETE /api/bot/me/premises/{premise_id}`. Backend: `status='inactive'`, обнуление зашифрованных полей и Blind Index контакта. Аудит: `action: premise_removed`. Остальные помещения и ответы сохраняются.
* **SR-BOT03-007:** Бот подтверждает пользователю выполнение каждого действия.

### 4. Сценарий использования
**Триггер:** Пользователь нажимает `[Мои данные]` или `/mydata`.
**Main Success Scenario (просмотр/редактирование):**
1. Бот показывает сводку данных (MY_DATA_VIEW).
2. Пользователь нажимает `[Изменить данные]` → переход в основной поток (PREMISES_OVERVIEW).
3. Пользователь обновляет ответы; данные сохраняются инкрементально.
**Alternative Flows:**
* **AF-1 (Нет записей):** MY_DATA_VIEW: «У нас пока нет ваших данных» + `[Заполнить анкету]`.
* **AF-2 (Полное удаление):** Пользователь нажимает `[Удалить все мои данные]` → CONFIRM_FORGET → подтверждение → soft delete → «Все ваши данные удалены» → IDLE.
* **AF-3 (Удаление телефона):** В CONTACT_MANAGE пользователь нажимает `[Удалить телефон]` → CONFIRM_DELETE → подтверждение → phone=null → CONTACT_MANAGE.
* **AF-4 (Удаление помещения):** В PREMISES_OVERVIEW/OFFER_PARKING_STORAGE/OFFER_MORE пользователь нажимает `[Убрать помещение]` → REMOVE_PREMISE → выбор → CONFIRM_REMOVE_PREMISE → `status='inactive'` + обнуление ПДн → PREMISES_OVERVIEW.

### 5. Модель данных
Поле `deleted_at` не используется. При удалении/забвении: `status='inactive'`, обнуление зашифрованных полей и Blind Index. Существующий фильтр `WHERE status IN ('pending','validated')` исключает неактивные записи. Различие между удалением помещения и забвением — в `action` аудит-лога (`premise_removed` / `forget`).

### 6. Интеграции и API

Все эндпоинты: авторизация `X-Bot-Token`.

**Получение всех данных:**
* **Method:** `GET /api/bot/me/data?telegram_user_id={id}` (переиспользуется из BOT-02).

**Полное удаление (забвение):**
* **Method:** `DELETE /api/bot/me/forget`
* **Request:**
```json
{
  "telegram_user_id": "123456789"
}
```
* **Response:** 200 OK `{"detail": "All data deleted"}`.
* Backend: все контакты по `telegram_id_idx` → `status='inactive'`, обнуление зашифрованных полей + Blind Index. Аудит: `action: forget`.

**Удаление телефона:**
* **Method:** `PATCH /api/bot/me/answers` с `"phone": null` (переиспользуется из BOT-02).

**Удаление помещения:**
* **Method:** `DELETE /api/bot/me/premises/{premise_id}` (переиспользуется из BOT-02). `status='inactive'`, обнуление ПДн.

### 7. Нефункциональные требования
* **Security:** Backend вычисляет `telegram_id_idx` (Blind Index) от переданного `telegram_user_id` и ищет контакты по нему. Чужие записи недоступны. После забвения Blind Index обнулён — повторный запрос вернёт пустой результат.
* **Audit:** Все действия логируются через BE-03. Различие: `action: premise_removed` (удаление одного помещения), `action: forget` (полное забвение), `action: phone_deleted` (удаление телефона). Записи содержат только факт действия и timestamp, без ПДн.

## BOT-04 — Синхронизация: единая БД, обновление по перезагрузке страницы

### 1. Архитектурный анализ (CoT)
Бот пишет через Backend API → данные сразу в общей БД PostgreSQL → веб-клиент получает актуальные данные при следующей загрузке/перезагрузке страницы. Автоматический polling / WebSocket / SSE не требуются.

### 2. Описание и цель
Данные из бота видны на шахматке, в кворуме и в админке при обновлении страницы. Специальных механизмов real-time синхронизации не требуется.

### 3. Функциональные требования
* **SR-BOT04-001:** Данные, созданные/обновлённые/удалённые через бота (BOT-02, BOT-03), должны сохраняться в той же БД, что и веб-анкеты.
* **SR-BOT04-002:** Веб-клиент при загрузке страницы или перезагрузке должен получать актуальные данные, включая записи из бота.
* **SR-BOT04-003:** Автоматическое обновление (polling, SSE, WebSocket) не требуется. Пользователь обновляет страницу для получения свежих данных.

### 4. Сценарий использования
**Триггер:** Пользователь внёс данные через бота; админ обновляет страницу в браузере.
**Main Success Scenario:**
1. Данные из бота сохранены в БД (через Backend API).
2. Админ обновляет страницу → видит обновлённые цвета на шахматке, актуальный кворум.

### 5. Модель данных
Не меняется; единая БД.

### 6. Интеграции и API
Существующие API: шахматка `GET /api/premises/chessboard`, кворум `GET /api/quorum`. Дополнительных эндпоинтов не требуется.

### 7. Нефункциональные требования
Нет дополнительных. Синхронизация обеспечивается единой БД.

### Фаза 1: Backend — миграция + таблицы

**Миграция Alembic 008:**
- `contacts.source` varchar(16) DEFAULT 'web' (web | telegram | admin)
- Таблица `premise_type_aliases` (premises_type, short_name, alias) + seed-данные

Новых колонок для telegram_user_id нет (используются существующие `telegram_id` + `telegram_id_idx`). Поле `deleted_at` не добавляется. Constraint на status не меняется (`inactive` уже допустим).

### Фаза 2: Backend — auth + resolver + router

**Авторизация бота:** `X-Bot-Token` (shared secret `BOT_API_TOKEN`, не Telegram-токен). Один бот для Login Widget и опроса (тот же `TELEGRAM_BOT_TOKEN`).

**BOT-01 Resolver:** загрузка `premise_type_aliases` из БД, нормализация ввода и номера помещения (а, б, в → a, b, b; 3В → 3b), парсинг тип+номер по синонимам, при отсутствии точного/LOWER-совпадения — fallback по нормализованному номеру (вариант A), fuzzy match (`rapidfuzz`), лог нераспознанных в `bot_unrecognized`.

**8 эндпоинтов `/api/bot/*`:** resolve-premise, premises (POST), premises/{id} (DELETE), me/answers (PATCH), me/data (GET), me/forget (DELETE), me/role (GET — роль пользователя для проверки суперадмина), admins-telegram-ids (GET — список telegram_id админов для рассылки им сообщений). Переиспользуют `submit_service`, `crypto`, `validators`, `room_normalizer`, `auth_telegram.get_admin_by_telegram_id`.

**Rate limit по `telegram_id_idx`** (in-memory, аналог IP-based в `rate_limit.py`).

**3 эндпоинта суперадмина:** CRUD `premise_type_aliases` + список `bot_unrecognized`.
