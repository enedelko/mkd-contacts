## BE-03 — Аудит-лог: INSERT, UPDATE, DELETE, SELECT

### 1. Архитектурный анализ (CoT)
Реализуется на Backend: перехват изменений (INSERT/UPDATE/DELETE) и при необходимости избранных SELECT (чтение контактов админом). Запись в отдельную таблицу или сервис логов в PostgreSQL. БТ требует логирование действий и чтения, IP-адреса.

### 2. Описание и цель
Централизованное логирование изменений данных и фактов чтения чувствительных данных для аудита и соответствия требованиям БТ.

### 3. Функциональные требования
* **SR-BE03-001:** Система должна логировать каждую операцию INSERT в таблицы, содержащие ПДн или критичные для кворума данные (контакты, помещения).
* **SR-BE03-002:** Система должна логировать каждую операцию UPDATE по таким таблицам с фиксацией старых и новых значений (или хешей/меток для ПДн).
* **SR-BE03-003:** Система должна логировать каждую операцию DELETE с идентификатором записи и контекстом.
* **SR-BE03-004:** Система должна логировать факт SELECT (чтения) записей с контактами при доступе администратора (какие записи, кто, когда).
* **SR-BE03-005:** В каждую запись лога система должна записывать: тип операции, сущность, идентификатор записи, идентификатор пользователя (если есть), IP-адрес, метка времени.
* **SR-BE03-006:** Система не должна записывать в лог сами значения ПДн (телефон/email в открытом виде); допускается идентификатор записи и тип поля.
* **SR-BE03-007:** Для связи событий в логах между собой без раскрытия ПДн система должна использовать Blind Index (phone_idx, email_idx, telegram_id_idx) или entity_id; в запись лога не попадают открытые контактные данные.
* **SR-BE03-008:** Система должна поддерживать фильтрацию аудит-лога по диапазону дат (from_date, to_date). Используется существующий индекс `ix_audit_log_created_at`.
* **SR-BE03-009:** Система должна поддерживать фильтрацию аудит-лога по entity_id (идентификатор записи).
* **SR-BE03-010:** Система должна отображать человекочитаемые подписи в аудит-логе: адрес помещения контакта (из таблицы premises) и ФИО администратора (из таблицы admins). Подписи подтягиваются через LEFT JOIN, без дополнительных запросов.
* **SR-BE03-011:** При UPDATE контакта в поле new_value система должна записывать список изменённых полей (без значений ПДн), например: `"phone,email,how_to_address"`.
* **SR-BE03-012:** В UI аудит-лога для записей entity_type=contact с числовым entity_id должна отображаться ссылка на карточку контакта (`/admin/contacts/{entity_id}`).
* **SR-BE03-013:** Все бот-действия (insert, premise_removed, bot_answers_update, forget) должны записывать telegram_user_id пользователя бота в поле user_id аудит-лога.
* **SR-BE03-014:** При user_id=NULL в UI аудит-лога система должна показывать «аноним» вместо прочерка. Для старых бот-записей без user_id система подтягивает telegram_id из контакта через decrypt() и возвращает как user_id_resolved для отображения ТГ-ссылки.

### 4. Сценарий использования
**Триггер:** Любое изменение или авторизованное чтение контактов.
**Main Success Scenario:**
1. Происходит INSERT/UPDATE/DELETE или чтение контакта админом.
2. Модуль аудита получает событие и записывает запись в таблицу логов.
**Alternative Flows:**
* **AF-1 (Переполнение лога):** Политика хранения (срок/размер) должна быть определена; при недоступности таблицы логов — логировать в fallback (файл) и не блокировать основную операцию.
* **AF-2 (Высокая нагрузка):** Запись в лог не должна существенно увеличивать время ответа API (асинхронная запись или быстрая вставка).

### 5. Модель данных
**Новые сущности:** audit_log: id, operation (INSERT/UPDATE/DELETE/SELECT), entity_type, entity_id, user_id, ip, timestamp, old_value_meta (без ПДн), new_value_meta (без ПДн).

### 6. Интеграции и API
Внутренняя реализация. API для просмотра логов — отдельный эндпоинт для админа (например, GET /api/admin/audit с фильтрами по времени, типу, пользователю).

### 7. Нефункциональные требования
* **Performance:** Асинхронная или быстрая запись в лог; индексы по timestamp, user_id, entity_type.
* **Security:** Доступ к просмотру логов только для администраторов; не хранить ПДн в логе.

## BE-04 — Защита: Rate Limiting (10 зап/час) и капча (Turnstile)

### 1. Архитектурный анализ (CoT)
Rate limiting реализуется на Backend (middleware или зависимость FastAPI) по IP. Капча — на Frontend виджет Turnstile; токен передаётся в теле запроса отправки анкеты и проверяется на Backend через API Turnstile. Оба механизма защищают от спама и ботов.

### 2. Описание и цель
Ограничение количества отправок анкет с одного IP (10 в час) и обязательная проверка капчи (Cloudflare Turnstile) при отправке формы.

### 3. Функциональные требования
* **SR-BE04-001:** Система должна ограничивать количество успешных запросов на создание анкеты (POST /api/submit) с одного IP-адреса до 10 в час.
* **SR-BE04-002:** При превышении лимита система должна возвращать HTTP 429 с заголовком Retry-After (или аналог) и не сохранять анкету.
* **SR-BE04-003:** Система должна принимать в запросе отправки анкеты токен капчи (Turnstile response token).
* **SR-BE04-004:** Система должна проверять токен капчи через API Turnstile перед сохранением анкеты.
* **SR-BE04-005:** Если проверка капчи не пройдена, система должна возвращать ошибку (например, 400) и не сохранять анкету.
* **SR-BE04-006:** Лимит 10 записей в час должен учитывать только успешно созданные анкеты (или 🤔 все попытки — уточнить; по БТ «лимит до 10 невалидированных записей» — допущение: 10 попыток/час с IP).

### 4. Сценарий использования
**Триггер:** Пользователь отправляет форму или злоумышленник выполняет массовые запросы.
**Main Success Scenario:**
1. Пользователь проходит капчу и отправляет форму.
2. Сервер проверяет капчу, проверяет счётчик по IP, создаёт запись и возвращает успех.
**Alternative Flows:**
* **AF-1 (Капча не пройдена):** Сервер возвращает 400, анкета не сохраняется.
* **AF-2 (Превышен лимит):** Сервер возвращает 429, клиент показывает сообщение о лимите.

### 5. Модель данных
Для rate limit: счётчики по IP хранятся в PostgreSQL (таблица или ключ-значение с TTL/очисткой по времени). Redis в текущий стек не входит; реализация — средствами БД.

### 6. Интеграции и API
* **Внешняя интеграция:** Backend вызывает API Turnstile для проверки токена (siteverify).
* **Поведение API:** POST /api/submit без валидного captcha_token или при 429 — см. разделы выше.

### 7. Нефункциональные требования
* **Security:** Rate limit по IP; при использовании прокси/общего IP возможны ложные срабатывания — документировать.
* **Масштабирование:** При нескольких инстансах backend счётчики должны храниться в общей БД (PostgreSQL), иначе лимит не общий.

## CORE-03 — Жизненный цикл контактов: ручной аудит после импорта

### 1. Архитектурный анализ (CoT)
Реестр не содержит ФИО; автоматическое отслеживание смены собственников по ФИО невозможно (04-import-chain, SR-CORE01-024). Актуализация контактов выполняется вручную администратором через VAL-01. В бэклоге — UI для ручного аудита после импорта реестра.

### 2. Описание и цель
Обеспечить администратору возможность после импорта реестра провести ручной аудит контактов: сопоставление записей по помещениям, установка статусов «валидирован» / «неактуальный» (VAL-01). Цель — актуализация базы при отсутствии ФИО в реестре.

### 3. Функциональные требования
* **SR-CORE03-001:** Система должна предоставлять UI-инструмент «Ручной аудит контактов после импорта реестра»: просмотр контактов по помещениям (по кадастровому номеру), фильтры по статусу (pending / validated / inactive), массовые или поштучные действия по смене статуса (интеграция с VAL-01).
* **SR-CORE03-002:** В интерфейсе аудита не должны отображаться ПДн в открытом виде без необходимости; при раскрытии — с записью в аудит (BE-03).
* **SR-CORE03-003:** Операции смены статуса выполняются через существующий API VAL-01 (PATCH /api/admin/contacts/{id}/status); аудит-лог фиксирует все изменения.
* **SR-CORE03-004:** Список контактов в интерфейсе аудита должен отображаться только после выбора подъезда (entrance); до выбора подъезда список не показывается. Выбор подъезда не должен реализовываться в виде выпадающего списка (select) — это лишний клик; предпочтительно ссылки списком.
* **SR-CORE03-005:** По клику по строке списка открывается просмотр контакта.
* **SR-CORE03-006:** После выбора подъезда система должна предоставлять поиск (фильтр) по номеру квартиры/помещения для сужения списка контактов.

### 4. Сценарий использования
**Триггер:** После загрузки реестра (CORE-01) администратор открывает раздел аудита контактов.
**Main Success Scenario:**
1. Админ выбирает подъезд (entrance); система сразу отображает список контактов помещений этого подъезда.
2. При необходимости админ вводит номер квартиры/помещения в поиск и сужает список.
3. Админ при необходимости кликает по строке списка и переходит к просмотру (детализации) контакта; при необходимости переводит контакт в «валидирован» или «неактуальный».
4. Система сохраняет изменения и пишет события в audit_log.
**Alternative Flows:**
* **AF-1 (Подъезд не выбран):** Список контактов не отображается; отображается только выбор подъезда или подсказка выбрать подъезд.
* **AF-2 (В подъезде нет контактов / нет невалидированных):** Отображается пустой список или сообщение.
* **AF-3 (Коллизия):** При наличии коллизий (CORE-01) отображать метаданные без раскрытия ПДн; решение — ручное.

### 5. Модель данных
Используются существующие таблицы contacts, premises; статусы по LOST-01. Дополнительных таблиц не требуется.

### 6. Интеграции и API
Используются GET /api/admin/contacts (или аналог с фильтрами по entrance, premise_id, status, номер помещения), PATCH /api/admin/contacts/{id}/status (VAL-01). Список контактов запрашивается с обязательным параметром entrance (подъезд); опционально — фильтр по номеру квартиры/помещения (premises_number или аналог). На том же экране могут быть доступны дополнительные фильтры ADM-02 (IP, диапазон дат) — см. [11-adm02-adm05-moderation-superadmin-audit.md](11-adm02-adm05-moderation-superadmin-audit.md). Новые эндпоинты — при необходимости расширения фильтров.

### 7. Нефункциональные требования
* **Security:** Доступ только для администратора; аудит всех просмотров и смен статусов (BE-03).

### 9. Доработки UI (уточнения реализации)

* **Аудит-лог (страница):** колонка «Пользователь» объединяет ФИО админа (из admins.full_name) и иконку ТГ-ссылки (`tg://user?id=...`); при user_id=NULL показывается «аноним»; для старых бот-записей без user_id — ТГ-ссылка по telegram_id контакта (user_id_resolved). Колонка «Запись» для контактов показывает адрес помещения и ссылку на карточку. Фильтры: по дате (с/по), entity_id, сущности, действию, пользователю. Словари ACTION_LABELS и ENTITY_TYPE_LABELS покрывают все реальные значения ([AuditLog.jsx](../../frontend/src/pages/AuditLog.jsx)).
* **Формат телефона:** форматирование на blur (вид +7 (XXX) XXX-XX-XX) в форме контакта и на форме подачи заявки (`formatPhoneOnBlur` из [phoneFormat.js](../../frontend/src/utils/phoneFormat.js)); улучшение читаемости ([AdminContacts.jsx](../../frontend/src/pages/AdminContacts.jsx), [Form.jsx](../../frontend/src/pages/Form.jsx)).
* **Список контактов:** колонки «ТГ» (иконка Telegram, ссылка по telegram_id или по номеру телефона), Email (mailto-ссылка); колонка ID убрана; русские подписи для ОСС (barrier_vote → ЗА/Против/Не определился, vote_format → Электронно/Бумага, registered_ed → Да/Нет); чередование строк (4 оттенка); короткий формат даты (дд.мм.гг); возврат «Назад к списку» с восстановлением подъезда через `location.state.fromEntrance` / `state.entrance` ([AdminContactsList.jsx](../../frontend/src/pages/AdminContactsList.jsx)).
* **Форма контакта:** иконка Telegram (ссылка в чат по telegram_id или по телефону) рядом с полем Telegram; кнопка «Назад к списку» с передачей подъезда в state (`state: { entrance: location.state?.fromEntrance }`) ([AdminContacts.jsx](../../frontend/src/pages/AdminContacts.jsx)).
